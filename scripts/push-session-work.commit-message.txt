Phase 1D Slice 1: sub portal magic-link funnel end-to-end

Backend (functions/src):
- bids/sendBidRequestRoute.ts: rewritten. Per-vendor 24-char base64url
  invite-token generation, /bidInviteTokens/{token} lookup docs, HTML email
  with View in Skyeline OS button, general (3-tier) + item bid types, legacy
  bidPackage fields preserved for the existing GC dashboard, Twilio init
  isolated so a bad SID can't kill email, skipDispatch flag for the
  bid-package consolidation flow, diagnostic logging on token writes.
- bids/bidTokenEndpoint.ts (NEW): public GET /api/bid-requests/by-token/:token
  for the magic-link landing page. FieldValue.serverTimestamp() inside the
  vendors[] array swapped to Timestamp.now() (Firestore rejects sentinels
  inside arrays). Improved 404 messages for withdrawn / deleted invitations.
- bids/bidPackageDispatchRoute.ts (NEW): POST /api/bid-packages/dispatch.
  Groups vendors across all trades in a package, sends ONE consolidated
  styled HTML email per vendor listing every trade they're invited to.
- bids/deleteBidRequestRoute.ts (NEW): POST /api/bid-requests/delete.
  Staff-only. Cascades: bidRequest doc + every bidInviteTokens/{token} +
  orphan parent bidPackages/{id} if no sibling trades remain.
- auth/postSignupLinkRoute.ts (NEW): POST /api/sub/post-signup-link.
  Email exact match → contact link; fuzzy match (name + phone + company)
  falls back to linkReviewQueue for staff resolution.
- auth/linkQueueResolveRoute.ts (NEW): POST /api/sub/link-queue/:id/resolve.
  Staff action: link to existing contact, create new contact, or no-match.
- auth/ensureContactAuth.ts + contactAuthBackfill.ts: pending_team ->
  pending_gc role normalization (D-001).
- index.ts: all new routes registered BEFORE the catch-all 404 (Express
  middleware runs in registration order; routes after the catch-all are
  unreachable - initial registration was after, which silently 404'd
  every new endpoint).

Client (client/src):
